# Phase 03.1: Codex Hook Support + Chinese Localization - Research

**Researched:** 2026-06-04
**Status:** Ready for planning

<summary>
## Research Summary

Phase 03.1 can reuse most of the existing hook ingestion architecture. Codex hooks expose a command-hook model close enough to the current Claude Code hook path: a hook command receives JSON over stdin, the payload includes session/project fields, and lifecycle events such as `SessionStart`, `PermissionRequest`, and `Stop` can be mapped into the current notifier taxonomy.

The main implementation risk is not IPC. It is keeping the data model provider-aware everywhere events flow: classification, session registry keys, TCP serialization/deserialization, notification cooldown keys, tray/history labels, and config installation. Chinese localization is best implemented as a small local formatting layer instead of scattering string literals through each module.
</summary>

<external_findings>
## External Findings

### Codex Hooks

Primary source: `https://developers.openai.com/codex/hooks`

Relevant facts for planning:

- Codex supports hook configuration at user and project levels, including `~/.codex/hooks.json` and `~/.codex/config.toml`.
- Codex runs command hooks by passing a JSON payload over stdin.
- Common Codex hook payload fields include `session_id`, `transcript_path`, `cwd`, and `hook_event_name`.
- Relevant Codex lifecycle events for this phase include `SessionStart`, `PermissionRequest`, and `Stop`.
- `PermissionRequest` is a natural semantic match for notifier `permission`.
- `Stop` is a natural semantic match for notifier `done`.
- `SessionStart` should update session/history state without notifying, per Phase 03.1 context D-08.

Implementation implication: add a provider-aware classifier rather than trying to force Codex payloads through Claude Code's `Notification` / `notification_type` matcher assumptions.
</external_findings>

<codebase_findings>
## Codebase Findings

### Current Event Model

- `src/notifier/core/events.py` has `EventCategory`, `SessionInfo`, `NotifierEvent`, and `classify_hook_event(raw, event_type)`.
- `SessionInfo` currently has `session_id`, `cwd`, and `project_name`.
- `NotifierEvent` currently does not carry provider.
- `classify_hook_event` currently assumes Claude Code event names and Claude notification matchers.

Planning implication: introduce a stable provider enum or string constants and make provider default to Claude Code for backward compatibility.

### Session Registry

- `src/notifier/server/tcp_server.py` has `SessionRecord` with `session_id`, `cwd`, `project_name`, `first_seen`, `last_seen`.
- `_registry_key(session_id, cwd)` currently returns `"{session_id}||{cwd}"`.
- `_handle_client` reconstructs `SessionInfo` from JSON and calls `_update_session`.

Planning implication: provider must survive `NotifierEvent.to_dict()` and TCP reconstruction, then be used in `_registry_key(provider, session_id, cwd)`.

### Hook CLI

- `src/notifier/cli/hook.py` reads stdin, classifies the raw payload, and forwards the event through `send_event_or_drop`.
- `process_hook_event(raw, event_type, ...)` has no provider parameter.

Planning implication: either add a provider argument to `process_hook_event` and the CLI command line, or infer provider from an explicit second CLI argument. Hook execution must remain lightweight.

### Config Installation

- `src/notifier/core/settings.py` currently handles Claude Code settings JSON only.
- Existing install behavior preserves unrelated Claude settings, removes notifier-owned entries, and deep-merges fresh hooks.
- `src/notifier/cli/config.py` exposes one `notifier-config install` command.

Planning implication: add Codex hook config helpers beside the existing Claude helpers. Do not replace the existing Claude install function without tests proving backward compatibility.

### User-Visible Copy

- Notification copy currently lives in `src/notifier/core/notify.py`.
- Tray menu labels are built in `src/notifier/tray/menu.py`.
- Event detail text is built in `src/notifier/tray/detail.py`.
- CLI output is printed in `src/notifier/cli/config.py`.

Planning implication: create a small `notifier.core.text` or equivalent module for Chinese labels, provider display names, category labels, and notification bodies. This reduces duplication and makes later language configuration possible without reworking every UI surface.
</codebase_findings>

<implementation_guidance>
## Implementation Guidance

### Provider Model

Use one small provider abstraction:

- `Provider.CLAUDE_CODE = "claude_code"`
- `Provider.CODEX = "codex"`

Keep display labels separate:

- `Claude Code`
- `Codex`

This preserves stable internal values while allowing Chinese text around them.

### Classifier Shape

Recommended function shape:

```python
def classify_hook_event(raw: dict, event_type: str, provider: Provider | str = Provider.CLAUDE_CODE) -> NotifierEvent:
    ...
```

Claude Code keeps current behavior. Codex uses a separate branch:

- `PermissionRequest` -> `EventCategory.PERMISSION`
- `Stop` -> `EventCategory.DONE`
- `SessionStart` -> non-notifying category choice:
  - Prefer category `idle` only if notification dispatch is updated to suppress `SessionStart`.
  - Better: keep category `idle` for existing taxonomy compatibility and add `_should_notify(event)` that suppresses provider/event combinations explicitly.
- Unknown Codex events -> `EventCategory.ERROR`

### Notification Suppression

Do not rely on category alone for notification decisions after provider support. Phase 03.1 needs event-level suppression:

- `SessionStart` from either provider updates session/history only.
- Codex unknown/error events are logged/history only.
- Codex has no synthesized idle notification in this phase.

### Codex Config Installation

Plan for a dedicated Codex settings helper:

- `CODEX_HOOKS_FILE = Path.home() / ".codex" / "hooks.json"`
- Optionally support a TOML helper only if existing config requires it.
- Build notifier-owned Codex entries for `SessionStart`, `PermissionRequest`, and `Stop`.
- Remove only entries whose hook command references `notifier.cli.hook` or `notifier-hook`.
- Preserve unrelated hooks and unrelated settings.
- Keep uninstall out of scope.

### Chinese Copy

Recommended labels:

- `permission` -> `需要授权`
- `idle` -> `等待输入`
- `done` -> `任务已完成`
- `error` -> `需要检查`
- no events -> `暂无事件`
- exit -> `退出`
- project path -> `项目路径`
- event type -> `事件`
- time -> `时间`

Keep provider names and technical categories available in detail/history, for example: `Codex - permission - 需要授权`.
</implementation_guidance>

<risks>
## Risks and Watch-outs

- **Provider missing in legacy events:** Existing tests and callers construct `SessionInfo` directly. Add defaults so old tests do not all need noisy updates unless the provider is part of the behavior under test.
- **Notification semantics:** `SessionStart` previously classified as `idle`. If `NOTIFY_CATEGORIES` stays category-only, `SessionStart` can still notify by accident. Add event-level notification eligibility tests.
- **Config overwrite risk:** Codex hook installation must prove it preserves unrelated hooks and settings, including existing user hook entries.
- **Encoding:** Chinese output must be written as UTF-8. Tests should assert expected Unicode strings in Python, not PowerShell console rendering.
- **Future Phase 4:** Use provider-aware session keys now so inactivity detection does not require a second registry migration.
</risks>
