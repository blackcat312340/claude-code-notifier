# Phase 03.1: Codex Hook Support + Chinese Localization - Context

**Gathered:** 2026-06-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Add first-class Codex support before abnormal-state detection. This phase extends the existing Claude Code notifier pipeline into a provider-aware event system, installs notifier-managed Codex hooks at user level, maps Codex lifecycle events into the existing reminder taxonomy, and switches the main user-visible text to concise Chinese copy.

**In scope:** provider-aware event/session model, Codex hook ingestion through the existing lightweight CLI-to-TCP path, user-level Codex hook installation, provider source display in notifications/history/details, and Chinese copy for notifications, tray menus, event details, and CLI output.

**Out of scope:** Codex hook uninstall and installed-status UI (Phase 5), configurable language switching, project-level Codex-only install mode, synthesized Codex idle detection, transcript parsing, and provider-specific abnormal-state heuristics (Phase 4).

</domain>

<decisions>
## Implementation Decisions

### Codex Integration Scope
- **D-01:** Treat Codex as a first-class provider, not as a Claude Code compatibility shim.
- **D-02:** Keep the visible app name `Claude Code Notifier` for now, but display event source as `Claude Code` or `Codex` in notifications, recent history, and event detail views.
- **D-03:** Add provider identity to both `NotifierEvent` and `SessionInfo`.
- **D-04:** Upgrade the session registry key from `(session_id, cwd)` to `(provider, session_id, cwd)` to avoid collisions between Claude Code and Codex sessions.

### Event Mapping Rules
- **D-05:** Preserve the existing four notifier categories: `permission`, `idle`, `done`, and `error`.
- **D-06:** Map Codex `PermissionRequest` to `permission`.
- **D-07:** Map Codex `Stop` to `done`.
- **D-08:** Treat Codex `SessionStart` as a session/history update only; it must not trigger a desktop notification.
- **D-09:** Do not synthesize Codex `idle` notifications when no explicit idle-equivalent hook exists. Phase 4 will use unified `last_seen` / inactivity detection for both providers.
- **D-10:** Unknown or unsupported Codex hook events become `error` records in history/logs, but do not fire desktop notifications.

### Chinese Localization Scope
- **D-11:** Default user-visible copy is Chinese-first for this phase. Do not add a language switch UI or config option yet.
- **D-12:** Cover notifications, tray menu labels, event detail popup text, and CLI output from install/config commands.
- **D-13:** Keep provider names and technical identifiers stable: `Claude Code`, `Codex`, `permission`, `idle`, `done`, `error`, hook event names, file paths, and CLI commands remain recognizable.
- **D-14:** Translate user-facing explanations into concise tool-style Chinese. Examples: `Codex 需要授权`, `任务已完成`, `等待输入`.

### Hook Installation Strategy
- **D-15:** Install Codex hooks by default at user level so all local Codex projects can be monitored, matching the product goal of all-project local monitoring.
- **D-16:** Manage only notifier-owned Codex hook entries. Preserve unrelated user Codex hooks and settings.
- **D-17:** `notifier-config install` should become the unified install command for both Claude Code and Codex hooks and should report each provider's install status.
- **D-18:** Do not implement Codex hook uninstall in this phase. Phase 5 remains responsible for safe hook lifecycle, uninstall, and visible installed-status reporting.

### the agent's Discretion
No major product decisions were delegated. Planners may choose implementation details such as exact enum/string names, helper function boundaries, and TOML/JSON parsing libraries, as long as the decisions above are preserved.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/notifier/core/events.py` - `EventCategory`, `SessionInfo`, `NotifierEvent`, and `classify_hook_event()` are the central model/classification seam. Add provider data here and split provider-specific classification cleanly.
- `src/notifier/core/session.py` - `extract_session()` and `project_name()` already derive session/project metadata from hook payloads. Extend extraction to accept provider and Codex payload fields.
- `src/notifier/server/tcp_server.py` - `NotifierServer.session_registry` and `_registry_key()` currently key by `session_id||cwd`. Update keying to include provider and keep `SessionRecord` provider-aware.
- `src/notifier/core/settings.py` - Existing Claude Code hook install/deep-merge code is the pattern for preserving unrelated user settings. Add Codex config handling beside it, not by overwriting user hooks.
- `src/notifier/cli/config.py` - `notifier-config install` is the user-facing install command that should become provider-unified and emit Chinese status output.
- `src/notifier/cli/hook.py` - Lightweight hook entrypoint already reads stdin, classifies, forwards via TCP, and should remain lightweight for Codex too.
- `src/notifier/core/notify.py` - Notification body building is where provider source and Chinese copy should be applied.
- `src/notifier/tray/menu.py` and `src/notifier/tray/detail.py` - Existing tray/history/detail surfaces should show provider and Chinese user-facing labels.

### Established Patterns
- Hook execution stays lightweight; heavier work remains in the resident notifier process.
- The local TCP NDJSON bridge is the shared ingestion path.
- Existing v1 categories are stable and should not be expanded for Codex unless a later phase explicitly changes taxonomy.
- Phase 3 stores recent history in memory, so Codex support should add source/provider fields without introducing persistent storage.

### Integration Points
- Claude Code user settings remain under `~/.claude/settings.json`.
- Codex user-level hooks should use the Codex user config layer, primarily `~/.codex/hooks.json` or `~/.codex/config.toml`.
- Hook commands for both providers should call notifier-managed Python entrypoints and forward normalized events to the same TCP server.

</code_context>

<specifics>
## Specific Ideas

- Notification title/body can include provider source without renaming the app, for example: title `Codex - notifier`, body `需要授权 - Bash`.
- Recent event labels can use concise Chinese plus stable technical category, for example: `Codex - permission - 需要授权`.
- CLI install output should be Chinese-first and provider-specific, for example: `Claude Code hooks: 已安装`, `Codex hooks: 已安装`.

</specifics>

<deferred>
## Deferred Ideas

- Configurable language selection (`zh-CN` / `en-US`) and following Windows system locale.
- Codex hook uninstall and visible installed-status reporting, to be handled in Phase 5.
- Unified inactivity/stall detection for Claude Code and Codex, to be handled in Phase 4.
- Project-level Codex-only install mode or provider-selective install flags.

</deferred>
