**Date:** 2026-06-04
**Phase:** 03.1-Codex Hook Support + Chinese Localization
**Areas discussed:** Codex integration scope, Event mapping rules, Chinese localization scope, Hook installation strategy

---

## Codex Integration Scope

| Option | Description | Selected |
|--------|-------------|----------|
| First-class provider | Model Claude Code and Codex as peer providers, so downstream history, tray, notifications, and anomaly detection can distinguish source. | yes |
| Minimum compatibility | Let Codex hook events enter the existing pipeline with the smallest possible model changes. | |
| Design only | Capture Codex support design but do not implement code in this phase. | |

**User's choice:** First-class provider.
**Notes:** The app name stays `Claude Code Notifier` for now, but notifications, history, and details show source as `Claude Code` or `Codex`. Provider identity belongs on both `NotifierEvent` and `SessionInfo`; the session key becomes `(provider, session_id, cwd)`.

---

## Event Mapping Rules

| Option | Description | Selected |
|--------|-------------|----------|
| Semantic equivalent mapping | Keep the existing four categories and map Codex events by meaning. | yes |
| Codex-specific categories | Add categories such as `codex_permission` or `codex_done`. | |
| Info-only | Record Codex events without triggering reminders. | |

**User's choice:** Semantic equivalent mapping.
**Notes:** Codex `PermissionRequest` maps to `permission`; `Stop` maps to `done`; `SessionStart` updates session/history only. Codex idle is not synthesized from `SessionStart` or message text; Phase 4 handles unified inactivity detection. Unknown Codex events become non-notifying `error` records in history/logs.

---

## Chinese Localization Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Notifications + tray + details + CLI | Cover main user-visible surfaces, including install/config command output. | yes |
| Notifications and tray only | Smaller scope; CLI and detail popup remain English. | |
| Notifications only | Fastest but too thin for meaningful Chinese support. | |

**User's choice:** Notifications + tray + details + CLI.
**Notes:** Default output is Chinese-first in this phase. No language switch UI or config setting yet. Keep provider names and technical identifiers recognizable while translating user-facing explanations. Copy style should be concise and tool-like.

---

## Hook Installation Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| User-level global config | Install Codex hooks for the current user so all local Codex projects can be monitored. | yes |
| Project-level config | Install only in the current repo's `.codex` layer. | |
| Both, default user-level | Add flexibility with more CLI/config complexity. | |

**User's choice:** User-level global config.
**Notes:** Preserve unrelated Codex settings/hooks and manage only notifier-owned entries. `notifier-config install` should install both Claude Code and Codex hooks and report each status. Codex uninstall is explicitly deferred to Phase 5 Safe Hook Lifecycle.

---

## the agent's Discretion

- Exact implementation details such as enum values, parser helpers, and file layout can be chosen by the planner.
- The planner must preserve first-class provider modeling, Chinese-first output, user-level Codex install, and non-destructive config merging.

## Deferred Ideas

- Language switch UI and locale/config-driven language selection.
- Codex hook uninstall and installed-status UI in Phase 5.
- Unified Codex/Claude inactivity detection in Phase 4.
- Provider-selective or project-local Codex install flags.
